## Phase 1: Ingestion & Static Analysis Foundation

**Goal:** Build the FastAPI service and Celery worker pipeline that accepts code input, runs SAST tools, deduplicates findings, and returns a normalized findings JSON. This is the skeleton everything else plugs into.

**UI hint:** no

**Requirements:**
- INGT-01: File upload endpoint
- INGT-02: Git URL clone and analyze
- INGT-03: Language detection
- INGT-04: Multi-file splitting
- INGT-05: Job status polling
- STAT-01: Semgrep with security rule sets
- STAT-02: Bandit for Python
- STAT-03: Deduplication by (file, line, CWE class)
- STAT-04: Normalized finding schema
- STAT-05: Filter INFO/non-security findings

**Plans:**
3/4 plans executed
2. Semgrep + Bandit integration with JSON parsing and normalized finding schema
3. Deduplication engine + findings store (SQLite)
4. Language detection + file splitter

**Success Criteria:**
1. POST /upload with a Python file returns a job_id within 500ms
2. GET /status/{job_id} returns `complete` after analysis finishes
3. GET /results/{job_id} returns findings with file, line, severity, rule_id, message fields
4. Semgrep runs with `p/security-audit` and `p/owasp-top-ten` — not `--config=auto`
5. Duplicate findings at the same (file, line, cwe_class) are merged into one record
6. INFO-level findings are excluded from results
7. A known-vulnerable Python file (e.g. SQL injection) produces at least one High/Critical finding

**Depends on:** None — standalone foundation

## Phase 2: CPG Integration (Joern)

**Goal:** Integrate Joern to generate Code Property Graphs and extract taint flow context for each static finding. CPG output is structured and attached to findings for downstream LLM and fusion use.

**UI hint:** no

**Requirements:**
- CPG-01: Joern CPG generation for submitted code
- CPG-02: Taint flow query per finding
- CPG-03: CPG context extracted and structured for prompts
- CPG-04: Graceful failure handling (pipeline continues if Joern errors)

**Plans:**
2/4 plans executed
2. CPG query engine — per-finding taint flow queries + result parser
3. CPG context serializer + fallback handler

**Success Criteria:**
1. Joern server starts and stays alive between analyses (no cold-start per file)
2. For a known SQL injection finding, Joern returns at least one taint path (source → sink)
3. CPG context is attached to each finding as `{taint_paths: [...], ast_excerpt: "..."}`
4. If Joern fails to parse a file, finding is marked `cpg_status: failed` but pipeline continues
5. CPG context is non-empty for at least 60% of High/Critical findings in test corpus

**Depends on:** Phase 1 (findings schema)

## Phase 3: RAG Knowledge Base

**Goal:** Build and persist a FAISS vector index over OWASP Top 10 and CWE corpus. For each finding, retrieve the top-3 most semantically relevant security documents to ground LLM reasoning.

**UI hint:** no

**Requirements:**
- RAG-01: Embed OWASP Top 10 into FAISS index
- RAG-02: Embed MITRE CWE list into FAISS index
- RAG-03: Persist FAISS index to disk
- RAG-04: Semantic search — top-3 docs per finding
- RAG-05: RAG docs injected into LLM prompt

**Plans:**
2/4 plans executed
2. Embedding pipeline — `all-MiniLM-L6-v2` → FAISS index build + persist/load
3. Retrieval service — query API that takes finding message → returns top-3 ranked docs

**Success Criteria:**
1. FAISS index loads from disk in < 2 seconds on startup
2. A finding with message "SQL injection via user input" retrieves CWE-89 in top-3 results
3. A finding with message "hardcoded password" retrieves CWE-798 in top-3 results
4. Retrieval latency < 50ms per finding
5. Index rebuild script works cleanly from scratch (for reproducibility)

**Depends on:** None — independent of Phases 1–2, parallelizable in development

## Phase 4: LLM Reasoning Layer

**Goal:** Send each enriched finding (static + CPG + RAG) to DeepSeek-Coder-V2 via Ollama. LLM returns structured analysis: explanation, root cause, confidence, and fix suggestion. Anti-hallucination and anti-prompt-injection controls are mandatory.

**UI hint:** no

**Requirements:**
- LLM-01: Send enriched finding to Ollama (DeepSeek-Coder-V2)
- LLM-02: Prompt instructs model to use only provided CWE context
- LLM-03: Code wrapped in delimiters (prompt injection prevention)
- LLM-04: Structured LLM output: explanation, root cause, fix, confidence (0–1)
- LLM-05: Graceful Ollama failure handling

**Plans:**
0/4 plans executed
2. LLM output parser — structured JSON extraction with validation
3. Async batch processor — queue findings, dispatch to Ollama workers, collect results

**Success Criteria:**
1. For a SQL injection finding enriched with CWE-89 RAG doc, LLM explanation mentions SQL injection and references the provided CWE context (not a hallucinated ID)
2. LLM confidence score is a float 0.0–1.0 in all responses
3. A code snippet containing `// IGNORE PREVIOUS INSTRUCTIONS` does not cause LLM to output anything other than vulnerability analysis
4. If Ollama times out, finding is marked `llm_status: failed` — pipeline does not crash
5. End-to-end latency (static → CPG → RAG → LLM) for a single finding < 60 seconds

**Depends on:** Phase 2 (CPG context), Phase 3 (RAG retrieval)

## Phase 5: Fusion / Decision Classifier

**Goal:** Train and deploy a scikit-learn ML classifier that combines static confidence, LLM confidence, and CPG graph features into a final verdict (true_positive / false_positive / uncertain). SHAP explanations surface which signals drove each decision.

**UI hint:** no

**Requirements:**
- FUSE-01: Feature vector construction per finding
- FUSE-02: GradientBoostingClassifier produces verdict
- FUSE-03: Trained with class_weight='balanced' on Big-Vul
- FUSE-04: SHAP values computed per finding
- FUSE-05: verdict + SHAP stored with finding

**Plans:**
0/4 plans executed
2. Classifier training pipeline — Big-Vul dataset prep + model training + serialization
3. Inference service + SHAP integration

**Success Criteria:**
1. Feature vector contains at minimum: static_confidence, llm_confidence, cpg_taint_depth, cwe_severity_score
2. Classifier achieves F1 ≥ 0.65 on Big-Vul validation split (must beat Semgrep-only baseline)
3. Classifier with `class_weight='balanced'` — false positive rate < 40% on validation set
4. SHAP values are computed and stored for every finding with a `true_positive` verdict
5. Model is serialized and loads in < 1 second on startup

**Depends on:** Phase 4 (LLM confidence scores)

## Phase 6: Patch Generation

**Goal:** For each confirmed vulnerability (true_positive verdict), LLM generates a corrected code snippet. System shows a diff and re-runs Semgrep to validate the fix eliminated the finding.

**UI hint:** no

**Requirements:**
- PATC-01: LLM generates corrected code for true_positive findings
- PATC-02: Diff view (original vs. patched)
- PATC-03: Semgrep re-run on patched code to validate fix
- PATC-04: Patch validation result stored with finding

**Plans:**
0/4 plans executed
2. Diff computation + Semgrep validation loop

**Success Criteria:**
1. For a known SQL injection, LLM generates a patch using parameterized queries
2. Semgrep re-run on the patched code returns 0 findings for the same rule
3. Diff view correctly shows added/removed lines
4. Patch validation result is one of: `fixed`, `still_vulnerable`, `unverified`
5. Patch generation runs only for `true_positive` findings — not for `false_positive` or `uncertain`

**Depends on:** Phase 4 (LLM), Phase 5 (fusion verdict)
